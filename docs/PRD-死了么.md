## 1. 产品概述

### 1.1 背景

中国独居人口已超过 1.25 亿，包括独居老人、独居青年、丁克、慢性病患者、户外/夜班工作者等。
独居人群面临的核心风险：突发疾病、意外摔倒、心理危机等情况下，**无人在第一时间发现**，从而延误救助。

海外类似产品：Snug Safety（美国）、Kitestring（已停服）、Lifeline24（英国）、SafetyNet（加拿大）。

### 1.2 产品定位

> 一款"每天一次轻量签到 + 异常自动预警 + 紧急 SOS"的生命守护工具。
>
> **当用户长时间未签到时，自动通知预设的亲友，第一时间发现异常。**

### 1.3 目标用户（Persona）

| 用户类型 | 画像 | 核心诉求 |
|---------|------|---------|
| **独居青年（25-40 岁）** | 一线城市打拼、单身、租房 | 给父母一份"我还活着"的安心 |
| **独居老人（60+ 岁）** | 子女不在身边，可能有慢性病 | 突发情况能被及时发现 |
| **慢性病患者** | 心脏病、癫痫、低血糖等 | 发病后能联动呼救 |
| **户外/夜班工作者** | 外卖员、滴滴司机、夜班护士 | 工作中遇险时能呼救 |
| **关怀者（亲属/子女）** | 想了解亲人状态但又不愿过度打扰 | 不打扰式监护 |

### 1.4 价值主张

- 对**当事人**：一键操作，不打扰、不焦虑、随手可签。
- 对**关怀者**：不必每天电话问候，异常时第一时间收到通知。
- 对**社会**：降低独居悲剧发生概率。

### 1.5 设计原则

1. **极简**：核心操作 ≤ 1 次点击。
2. **不打扰**：不强推社交、不强推内容。
3. **隐私优先**：位置信息默认不上传，仅 SOS 时启用。
4. **温暖文案**：避免黑色暗示，使用积极语言（"今天也要好好的" 而非 "你死了吗"）。

## 3. 功能模块

### 3.1 功能总览

```
死了么 App
├── ① 签到模块（核心）
│   ├── 一键签到
│   ├── 签到日历（连续打卡）
│   └── 自定义签到周期（24/48/72h）
├── ② 预警模块
│   ├── 超时检测
│   ├── 多级预警（站内 → 短信 → 电话 → 紧急联系人）
│   └── 误报申诉/取消预警
├── ③ 紧急联系人
│   ├── 联系人管理（最多 5 人）
│   ├── 关怀者反向查看
│   └── 关系验证
├── ④ 穿戴设备联动
│   ├── Apple Watch / 华为/小米手环
│   ├── 心率/血氧异常触发
│   ├── 跌倒检测联动
│   └── 久坐不动检测
├── ⑤ SOS 一键呼救
│   ├── 长按 3s 触发
│   ├── 倒计时取消（防误触）
│   ├── 自动定位上报
│   └── 同时拨打 120/110（可选）
├── ⑥ 账号与隐私
│   ├── 手机号登录
│   ├── 紧急医疗信息
│   └── 数据加密 / 注销
└── ⑦ 设置
    ├── 静音时段
    ├── 出差模式
    └── 通知偏好
```

### 3.2 模块一：签到（Check-In）

#### 3.2.1 功能描述
用户每天打开 app，主页面正中心显示一个大按钮，点击即完成签到。

#### 3.2.2 详细需求

| 需求 ID | 需求描述 | 优先级 |
|---------|---------|--------|
| F1-01 | 主页显示一个大圆形按钮，文案"我今天还在"，点击后签到 | P0 |
| F1-02 | 签到成功显示动画 + 暖心文案（如"今天也是元气满满的一天"） | P1 |
| F1-03 | 支持指纹/Face ID 二次确认（防小孩误点） | P2 |
| F1-04 | 签到日历：显示连续打卡天数、本月打卡情况 | P1 |
| F1-05 | 自定义签到周期：24h（默认）/48h/72h | P0 |
| F1-06 | 出差模式：临时关闭预警 1-30 天，需二次确认 | P0 |
| F1-07 | 签到提醒推送：每日定时提醒 + 临近超时强提醒 | P0 |

### 3.3 模块二：预警（Alert）

#### 3.3.1 触发逻辑

```
T = 用户设定的签到周期（默认 24h）
W = 预警宽限期（默认 12h）

┌──────────────┬──────────────┬──────────────┐
│ 距上次签到 < T │ T ~ T+W      │ > T+W        │
│ 状态：正常    │ 状态：临近超时 │ 状态：超时    │
│ 无动作       │ 强提醒用户本人 │ 触发预警流程  │
└──────────────┴──────────────┴──────────────┘
```

#### 3.3.2 多级预警流程

```
第 1 级：站内推送 + 弹窗提醒（用户本人）
   ↓ 30 分钟未响应
第 2 级：短信 + 电话语音呼叫（用户本人）
   ↓ 30 分钟未响应
第 3 级：通知第 1 紧急联系人（短信 + app 推送）
   ↓ 1 小时未响应
第 4 级：通知所有紧急联系人 + 显示用户最后已知位置
   ↓ 关怀者可在 app 内点击"已确认安全"取消预警
```

### 3.4 模块三：紧急联系人

| 需求 ID | 需求描述 | 优先级 |
|---------|---------|--------|
| F3-01 | 添加紧急联系人（手机号 + 关系备注），最多 5 人 | P0 |
| F3-02 | 联系人需扫码或点击邀请链接确认建立关系 | P0 |
| F3-03 | 联系人可下载 app 或仅以短信方式接收预警 | P0 |
| F3-04 | 设置预警接收顺序（第 1、第 2、第 3...） | P1 |
| F3-05 | 关怀者视图：查看被关怀人的签到状态 | P0 |

### 3.5 模块四：穿戴设备联动

支持 Apple Watch（HealthKit）、华为运动健康、小米手环、Garmin。

| 能力 | 触发条件 | 动作 |
|------|---------|------|
| **被动签到** | 设备检测到当日有效活动 | 视为已签到，免手动 |
| **跌倒检测** | Apple Watch / 华为手环跌倒事件 | 立即触发 SOS（带 60s 倒计时） |
| **心率异常** | 静息心率 > 130 或 < 40 持续 5 分钟 | 推送提醒 + 询问是否触发 SOS |
| **血氧异常** | SpO2 < 90% 持续 3 分钟 | 提醒用户 + 关怀者通知（可关闭） |

### 3.6 模块五：SOS 一键呼救

| 入口 | 操作 |
|------|------|
| App 主页 | 长按红色 SOS 按钮 3 秒 |
| 锁屏 | iOS Action Button / Android 快捷方式 |
| 手表 | 长按表冠 / 自定义按钮 3 秒 |
| 跌倒检测 | 自动触发（60 秒倒计时） |
| 语音助手 | "嘿 Siri，呼救" / "小爱同学，紧急呼救" |

### 3.7 模块六：账号与隐私

- 手机号 + 短信验证码登录
- 第三方登录：Apple / 微信 / 华为账号
- 紧急医疗信息：血型、过敏、慢性病、用药
- 数据传输 HTTPS / 存储 AES-256 加密
- 通过《个人信息保护法》合规自评

## 4. 非功能性需求

| 类别 | 要求 |
|------|------|
| **可靠性** | 预警调度任务 99.95% 可用，单点故障不丢预警 |
| **延迟** | SOS 触发到联系人收到通知 ≤ 10 秒 |
| **离线** | 离线状态下 app 可缓存签到，恢复网络后自动上报 |
| **电量** | 后台耗电 < 同类社交 app 平均水平 |
| **安全** | 所有 API HTTPS / TLS 1.3，敏感字段加密存储 |
| **隐私** | 通过 GB/T 35273 个人信息安全规范、PIPL 合规 |
| **可访问性** | 支持大字体、VoiceOver / TalkBack（老年人友好） |

## 5. MVP 范围（第一版上线）

### ✅ MVP 必备（2-3 个月）
- 手机号登录
- 一键签到 + 自定义周期
- 紧急联系人管理（最多 3 人）
- 多级预警（推送 + 短信，先不做电话）
- SOS 一键呼救（长按 3s + 倒计时 + 通知联系人）
- 紧急医疗信息卡片
- iOS + Android 双端

### 🟡 V1.5（上线后 1-2 个月）
- 关怀者视图（家人侧 app 体验）
- 出差模式
- Apple Watch 联动（先做 Apple，覆盖度高）

### 🟢 V2.0（半年内）
- 华为/小米手环联动
- 跌倒检测联动 SOS
- 自动拨打 120 / 110
- 预警事件回放与统计

## 7. 风险与合规

| 风险 | 应对 |
|------|------|
| **误报扰民** | 多级预警 + 误报率监控 + 用户可随时调整周期 |
| **漏报伤命** | 双链路通知（推送 + 短信）+ 监控告警 |
| **隐私担忧** | 默认不采集位置、健康数据本地处理、PIPL 合规 |
| **品牌名称争议** | 建议正式上线时考虑双品牌：技术品牌"StillHere/还在"，营销话题"死了么" |
| **法律责任** | 用户协议明确：本产品为辅助工具，不替代专业医疗/急救服务 |
